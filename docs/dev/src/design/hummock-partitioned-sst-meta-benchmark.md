# Hummock Partitioned SST Meta Benchmark Plan

This document records the benchmark-only implementation scope for the partitioned SST metadata prototype. The RFC design remains in `hummock-partitioned-sst-meta.md`; this file exists to preserve benchmark context and avoid mixing prototype shortcuts into the production design.

## Goal

Build a benchmarkable v3-only prototype for:

```text
MetaPartitionIndex
  -> MetaShard { Vec<BlockMeta>, one plain group filter for N data blocks }
```

The benchmark should answer whether co-located `MetaShard` can reduce metadata bytes and cache footprint without increasing object-store metadata request count on the steady-state point-read path.

## Assumptions

- The benchmark workload uses only v3 SSTs generated by a dedicated point-get benchmark or test harness.
- v1/v2 SST compatibility is not required for the first benchmark implementation.
- Data blocks continue to use the existing data block cache.
- The first shard size candidates are 8 and 16 data blocks per `MetaShard`.
- `MetaShard.filter` is one plain group filter for all keys in the shard, not N per-block filters.
- Fast compaction must support v3 whole-shard raw-copy for the benchmark. Partial-shard filter reuse is out of scope and must fall back to decoded/rewrite output.
- Scan prefetch and additional filter-builder memory optimization are deferred until benchmark data shows they are needed.

## Prototype Switch

The first prototype can be enabled in a local benchmark or data-correctness run by setting:

```text
RW_SSTABLE_META_SHARD_BLOCK_COUNT=8
```

Use `16` for the second shard-size run. Leaving the variable unset keeps the legacy v2 write path.

For unit tests or dedicated harnesses, the same setting can be applied directly through:

```rust
SstableBuilderOptions {
    partitioned_meta_block_count: Some(8),
    ..Default::default()
}
```

This is a prototype-only switch, not the final production configuration surface. Use it only in controlled local benchmark/e2e runs until mixed-version compatibility, compaction, validation, and backward-iterator paths are completed.

## Required Prototype Behavior

### Write Path

The builder should write:

```text
data blocks
MetaShardBody[]
MetaPartitionIndexBody
checksum | version = 3 | magic
```

`MetaShardBody`:

```text
u32 block_meta_count
repeated BlockMeta
u32 filter_len
bytes filter_payload
```

`MetaPartitionIndexBody`:

```text
u32 estimated_size
u32 key_count
bytes smallest_key
bytes largest_key
u32 block_count
u32 shard_count
u32 filter_type
u32 shard_policy
repeated MetaShardDesc
```

`MetaShardDesc`:

```text
u32 first_block_idx
u32 block_count
bytes smallest_key
u64 offset
u32 len
u64 checksum
```

### Read Path

The point-read path should be:

```text
SstableInfo key-range/table-id prune
  -> load/cache MetaPartitionIndex
  -> locate MetaShard by key
  -> load/cache MetaShard
  -> check MetaShard.filter
  -> if filter positive, locate data block from MetaShard.block_metas
  -> read data block
```

The prototype must not decode all `MetaShard` bodies and reconstruct a full legacy `SstableMeta` before serving reads.

## Implementation Alignment Checklist

Run this checklist after each prototype code change. The goal is to keep correctness gaps separate from later I/O optimization work.

| Area | Required check | Evidence |
|---|---|---|
| Key semantics | Shard/block lookup follows the legacy `FullKey` comparison semantics; point-get shard-filter pruning expands candidates by `UserKey` so same-user-key versions across shards cannot be false-negative pruned. | Focused point-get tests with multi-version keys crossing shard boundaries; review `MetaPartitionIndex` and iterator locate helpers. |
| Forward iterator | v3 forward scans lazily load `MetaShard` bodies, read data blocks by shard-local `BlockMeta`, and enforce `table_id` range after seek/next. | `forward_sstable_iterator::tests::test_read_table_id_range` with v3 shard size. |
| Backward iterator | v3 backward scans do not index the empty legacy `meta.block_metas`; they lazily load shards and enforce `table_id` range after rewind/seek/next. | `backward_sstable_iterator::tests::test_read_table_id_range` with v3 shard size. |
| Scan filter pruning | v3 scans must not use the empty legacy global filter as a false-negative source. | Unit assertion that v3 `Sstable::may_match_hash` remains conservative for scan pruning. |
| Point-get filter pruning | v3 point reads check shard group filters and return `None` only when all candidate shard filters are negative. | `test_partitioned_meta_point_get` and same-user-key multi-version shard-boundary regression. |
| Normal compaction | Compaction input iterators load v3 block metadata from `MetaShard` bodies instead of reading the empty legacy `meta.block_metas`. | Storage full unit tests and e2e cases that trigger manual Hummock compaction. |
| Compaction split generation | Split generation uses v3 shard-loaded block metadata when preloading SST metadata. | `cargo test -p risingwave_storage --lib` plus code review of `generate_splits`. |
| Cache refill optimization | Data-cache refill must not panic or read wrong ranges for v3 SSTs. It can be conservatively disabled until a shard-aware refill path is implemented. | Code review of `event_handler::refiller` and full storage unit tests. |
| Fast compaction | v3 fast compaction must raw-copy only whole shard-sized runs and reuse the shard filter payload only for that whole-shard copy. Partial shards fall back to decoded/rewrite output and rebuilt filters. | Focused UT proves raw-copy block count and shard count for v3 whole-shard input; no partial-shard filter reuse. |
| Storage unit gate | Storage crate compiles and full lib tests pass after each meaningful change. | `cargo check -p risingwave_storage --lib`; `cargo test -p risingwave_storage --lib -- --test-threads=1`. |
| E2E data gate | At least one clean v3 data-correctness run covers flush/backfill/compaction; avoid treating invalid all-glob SLT runs as evidence. | `RW_SSTABLE_META_SHARD_BLOCK_COUNT=8 ./risedev d`; selected SLT runs such as `backfill_order_control.slt` and broad streaming subsets. |

## Benchmark Metrics

Collect or derive:

| Metric | Source | Purpose |
|---|---|
| `MetaPartitionIndex` bytes per SST | `state_store_partitioned_meta_counts{type="index_read_bytes"}` on index cache miss, plus object dump for per-SST size | Verify index remains small |
| `MetaShard` bytes distribution | `MetaShardDesc.len` or `state_store_partitioned_meta_counts{type="shard_read_bytes"}` | Compare shard size 8 vs 16 |
| metadata range-read count per point get | `index_cache_miss + shard_cache_miss` in `state_store_partitioned_meta_counts` | Verify steady-state request count |
| metadata bytes read per point get | `index_read_bytes + shard_read_bytes` in `state_store_partitioned_meta_counts` | Compare against old full-tail metadata |
| data block read count per point get | existing `state_store_sst_store_block_request_counts` data labels | Ensure data path is unchanged |
| shard filter negative / positive count | `shard_filter_negative` / `shard_filter_positive` in `state_store_partitioned_meta_counts` | Check group filter usefulness |
| index cache hit/miss | `index_cache_total` / `index_cache_miss` in `state_store_partitioned_meta_counts` | Confirm index stays hot |
| shard cache hit/miss | `shard_cache_total` / `shard_cache_miss` in `state_store_partitioned_meta_counts` | Measure metadata locality |
| p50/p99 point-get latency | benchmark harness latency | End-to-end benchmark signal |

## Success Criteria

- With `MetaPartitionIndex` cached, one candidate SST requires at most one metadata range read before a data block read.
- Metadata bytes read per point get are lower than the current full-meta path.
- Point-get latency does not regress enough to offset metadata byte savings.
- The result includes enough metrics to compare shard size 8 vs 16.

## Deferred Items

- v1/v2 and v3 mixed-version read path.
- Production migration and feature gating.
- Full cluster-wide enablement after compaction, validation, and backward-iterator paths have v3 handling.
- Partial-shard raw-copy/filter reuse.
- Scan prefetch and batched `MetaShard` reads.
- Advanced cache admission or priority policy beyond the minimal prototype. The current prototype keeps `MetaPartitionIndex` in the existing meta cache and uses a fixed-size in-memory `MetaShard` cache.
- Optimizing group-filter builder memory.

## Open Benchmark Questions

- Should `MetaShardDesc.smallest_key` be stored redundantly, or should the index use a smaller separator/index representation?
- Does shard size 8 or 16 produce a better tradeoff between filter precision, metadata bytes, and request count?
- How often does the shard filter return false positive for point reads?
- Is `MetaShard` cache locality good enough to justify not embedding all block metadata in `MetaPartitionIndex`?
